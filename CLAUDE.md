Qt6 GUI client only — connects to a showeq-daemon over WebSocket. No pcap, no caps needed; AppImage's FUSE/squashfs file-cap stripping is therefore irrelevant here (it would matter for the daemon).
AppImage build: `cmake --build build --target appimage` produces `build/showeq-qt-*.AppImage`. Driver is `packaging/build-appimage.sh`; it fetches linuxdeploy + linuxdeploy-plugin-qt on demand into `build/tools/`.
CI publishes a rolling `continuous` GitHub prerelease on push to main, gated on ubuntu/fedora/windows/appimage jobs. Stable URL: `https://github.com/showeq-unofficial/showeq-qt/releases/download/continuous/showeq-qt-continuous-x86_64.AppImage`.
AppImage CI container is `debian:bookworm-slim` — do NOT use `ubuntu:22.04` (jammy lacks `qt6-websockets-dev`; only in noble/24.04+). Bookworm has it and keeps glibc at 2.36 for broad distro coverage. Set `APPIMAGE_EXTRACT_AND_RUN=1` in GHA containers (no FUSE).
`vcpkg-configuration.json` baseline rots: Windows breaks deterministically when MSYS2 retires upstream packages (every mirror 404s on the pinned version). Fix is bumping the SHA to current `microsoft/vcpkg` master.
First Windows CI run after a vcpkg baseline bump rebuilds Qt6 + protobuf 6.x + abseil from source — expect ~2.5–3h on standard `windows-latest` runners. Subsequent runs hit GHA cache (<5 min). Don't cancel mid-build; partial vcpkg work doesn't survive cancel.
Linux jobs use system Qt6 + system protobuf (versions vary by distro); Windows uses vcpkg. `seq-proto` regenerates `.pb.cc` per build, so cross-platform protobuf-version drift is cosmetic.
